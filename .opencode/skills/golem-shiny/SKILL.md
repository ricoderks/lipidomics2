# Skill: Building Production-Ready Shiny Apps with {golem}

## Overview

`{golem}` is an opinionated framework for building production-grade Shiny applications. The core philosophy is that **a golem app IS an R package** — this means all package development best practices (documentation, testing, dependency management, versioning) apply directly to your Shiny app.

Key prerequisites:
- Familiarity with R package development (DESCRIPTION, NAMESPACE, roxygen2)
- Basic knowledge of Shiny modules (recommended, not mandatory)

---

## Installation

```r
# Stable from CRAN
install.packages("golem")

# Development version from GitHub
remotes::install_github("Thinkr-open/golem")
```

## Project Structure

### Creating a new golem app

**Via RStudio:** File > New Project > "Package for Shiny App Using golem"

**Via console:**

The name of package is checked if it is correct according to CRAN, if incorrect stop here and ask the user.

```r
golem::create_golem(path = "path/to/myapp", check_name = TRUE)
```

### Generated project layout

```
myapp/
├── DESCRIPTION              # Package metadata & dependencies
├── NAMESPACE                # Auto-generated by roxygen2
├── R/
│   ├── app_config.R         # Reads inst/golem-config.yml
│   ├── app_server.R         # Top-level server function
│   ├── app_ui.R             # Top-level UI function
│   └── run_app.R            # Entry point to launch the app
├── dev/
│   ├── 01_start.R           # One-time project setup commands
│   ├── 02_dev.R             # Day-to-day development commands
│   ├── 03_deploy.R          # Deployment preparation commands
│   └── run_dev.R            # Script to run the app in development
├── inst/
│   ├── app/
│   │   └── www/             # Static assets (CSS, JS, images)
│   │       └── favicon.ico
│   └── golem-config.yml     # golem configuration file
└── man/
    └── run_app.Rd
```

The `dev/` scripts are **notebooks**, not production code. Run them interactively to set up and develop your project.

## Phase 1: Project Initialisation (`dev/01_start.R`)

Run these commands **once** at the start of the project.

### Fill the DESCRIPTION file

If the user hasn't specified, ask for the users:

- given and family name 
- email address
- github repro url

```r
golem::fill_desc(
  pkg_name = "myapp",
  pkg_title = "My Shiny Application",
  pkg_description = "A Shiny app that does something useful.",
  authors = person(
    given = "First",
    family = "Last",
    email = "you@example.com",
    role = c("aut", "cre")
  ),
  repo_url = "https://github.com/you/myapp",
  pkg_version = "0.0.0.9000"
)
```

### Set up common project files

If the use hasn't specified yet, ask the user which license should be used:

- MIT -> usethis::use_mit_license()
- GPL -> usethis::use_gpl_license()
- AGPL -> usethis::use_agpl_license()
- LGPL -> usethis::use_lgpl_license()
- Apache -> usethis::use_apache_license()
- CC0 -> usethis::use_cc0_license()
- CCby -> usethis::use_ccby_license()
- proprietary -> usethis::use_proprietary_license()

Use the appropriate function from the package usethis to add the correct license. 

```r
usethis::use_readme_rmd(open = FALSE)
usethis::use_code_of_conduct()
usethis::use_lifecycle_badge("Experimental")
usethis::use_news_md(open = FALSE)
```

### Initialise tests

```r
golem::use_recommended_tests()
```

### Add recommended dependencies

Adds shiny, DT, attempt, glue, htmltools, and golem to DESCRIPTION:

```r
golem::use_recommended_deps()
```

### Add utility helpers

```r
golem::use_utils_ui(with_test = TRUE)
golem::use_utils_server(with_test = TRUE)
```

This creates `R/golem_utils_ui.R` and `R/golem_utils_server.R` with commonly needed helper functions.

### Launch the app for the first time

```r
golem::run_dev()
```

## Phase 2: Day-to-Day Development (`dev/02_dev.R`)

### Running the app

Use `dev/run_dev.R` to launch during development:

```r
golem::run_dev()
```

### Adding Shiny modules

Modules are the primary way to organise a golem app. Each module gets its own file:

```r
golem::add_module(name = "data_import")
# Creates R/mod_data_import.R
```

The generated file contains scaffolded UI and server functions:

```r
mod_data_import_ui <- function(id) {
  ns <- NS(id)
  tagList(
    # UI elements here
  )
}

mod_data_import_server <- function(id) {
  moduleServer(id, function(input, output, session) {
    # Server logic here
  })
}
```

Wire modules into `app_ui.R` and `app_server.R` using the copy-paste comments at the bottom of each generated file.

### Adding helper function files

```r
golem::add_fct("helpers")   # Creates R/fct_helpers.R  — business logic
golem::add_utils("helpers") # Creates R/utils_helpers.R — small utility functions
```

Convention: `fct_` for feature-level functions, `utils_` for general-purpose helpers.

### Creating functions

For each new function a description according to the roxygen2 package should be created. This should
always contain **@noRd**. **No** function should contain **@export**, except the run_app() function. 
All parameters of the function should be described with **@param**.
Make sure when creating new functions which use functions from packages other then base package to 
call these functions as pkg::fn(). For each package and function use add a **@importFrom <function name>**.

### Managing dependencies

Use `{attachment}` to auto-detect and update DESCRIPTION from your `@import`/`::` usage:

```r
attachment::att_amend_desc()
```

### Adding external assets (CSS, JS, Sass)

```r
golem::add_css_file("custom")        # inst/app/www/custom.css
golem::add_sass_file("custom")       # inst/app/www/custom.scss
golem::add_js_file("script")         # inst/app/www/script.js
golem::add_js_handler("handlers")    # JS with Shiny custom handler skeleton
```

Download from a URL:

```r
golem::use_external_css_file(url = "https://...", name = "bootstrap-custom")
golem::use_external_js_file(url = "https://...", name = "chart-lib")
```

All assets in `inst/app/www/` are automatically linked via `golem_add_external_resources()` in `app_ui.R`. Reference them in the app with the `www/` prefix:

```r
tags$img(src = "www/logo.png")
```

### Adding data

```r
usethis::use_data_raw()  # Creates data-raw/ for data preparation scripts
```

### Adding tests

```r
usethis::use_test("my_module")
```

### Code finished

When you finished generating the code **always** run the functions below.

- update NAMESPACE file
- update DESCRIPTION file
- check if every requirement is met for an R package

```r
devtools::document(roclets = c('rd', 'collate', 'namespace'))
attachment::att_amend_desc()
devtools::check()
```

The devtools::check() should result in 0 errors, 0 warnings and 0 notes. There is one exception for notes, sometimes it is not possible to get the current time.

## Key Conventions & Best Practices

| Convention | Detail |
|---|---|
| One module per feature | Keep modules small and focused |
| `fct_` prefix | Feature-specific business logic functions |
| `utils_` prefix | Generic, reusable helper functions |
| `mod_` prefix | All module files and function names |
| Dev scripts are notebooks | `dev/*.R` are for interactive use, not sourced in production |
| Everything is a package | Use `devtools`, `roxygen2`, `testthat` as normal |
| `golem-config.yml` | Central configuration; read via `get_golem_config()` |

## Common Workflow Summary

```
1. golem::create_golem("myapp")          # scaffold the project
2. Run dev/01_start.R interactively      # configure project once
3. golem::run_dev()                      # verify app runs
4. golem::add_module("feature_x")        # add modules for each feature
5. attachment::att_amend_desc()          # keep DESCRIPTION up to date
6. usethis::use_test("feature_x")        # write tests
7. devtools::check()                     # run R CMD check
```
